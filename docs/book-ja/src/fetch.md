# `fetch` — 消費せずに収集する

通常のエントリはディレクトリ内のパスを「消費」する。消費済みのパスは他のエントリと重複してマッチできない。しかし同じパス群を複数の視点から検証したい場合がある。そのときは `fetch:` を使う。

`fetch:` は `::` 配下のエントリにマッチするパスを収集するが、それ自体はディレクトリの消費を行わない。収集した結果を `id:` で束ね、後続の別ルールでも同じパスに対して検証を適用できる。

## 解説

この例では `items/` 内のパスを次の4段階で処理する。

### 1. ディレクトリ名を消費せずに収集する

```yaml
- fetch:
  id: dirs
  ::
    - dir:
        regex: '^(?<n>[a-z])$'
      ::
        - file: .keep
```

`fetch:` ブロックは `a/`、`b/`、`c/` などのディレクトリをキャプチャ `name` で収集するが、これらのパスを消費しない。後続のエントリが同じパスをあらためて消費できる。

### 2. 各ディレクトリを `pair` または `single` に分類する

収集した `dirs` を `for` でイテレートし、各ディレクトリが同名のテキストファイルと対になっているか（`pair`）、どちらか一方だけか（`single`）を `one_of` で判定する。分類結果全体を `id: classified` として束ねる。

### 3. `pair` ディレクトリの検証

`pair/` ディレクトリでは `${for.classified}` を `for` でイテレートし、`match` で `pair` タグのエントリのみ展開する。`pair` に分類されたパスについてはディレクトリとテキストファイルの両方を要求し、`single` に分類されたパスについては何も要求しない。

### 4. `single` ディレクトリの検証

`single/` ディレクトリでは同様に `${for.classified}` をイテレートし、`single` タグのエントリについてはディレクトリまたはテキストファイルのどちらか一方を要求する。

`fetch:` を使うことで「同じディレクトリ群を最初に名前だけ把握し、その後で詳細な分類と検証を行う」という2段階の処理が可能になる。先に `fetch` で素性を確認してから、後続の `for` で本格的な消費と検証を行うパターンである。
