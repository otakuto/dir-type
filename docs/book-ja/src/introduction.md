# はじめに

## コードには型があるのに、ファイルシステムには型がない

Rust や TypeScript を書いているとき、型が合わなければコンパイラがすぐに教えてくれる。`String` を渡すべき場所に `i32` を渡せば一瞬で検出される。型の規約は暗黙知にならない。

ではディレクトリ構造はどうだろうか。

```text
src/
  components/
    Button/
      index.tsx
      Button.test.tsx      ← 規約どおり。コンポーネントとテストが揃っている
    Card/
      index.tsx            ← テストがない。書き忘れ?
    Modal/
      index.tsx
      Modal.spec.tsx       ← 命名がずれている (.test.tsx のはずが .spec.tsx)
  hooks/                   ← 各ページ配下に置く規約のはずが、トップに生えている
    graphql/
    user/
      useUser.ts
  pages/
    home.tsx
    Dashboard/
      index.tsx
    settings_page.tsx
  utils.ts
  helpers.ts               ← utils.ts と役割が被っていないか?
```

ソースコードとテストの対応も崩れやすい。`auth.ts` のテストが `auth.spec.ts` だったり `auth.test.ts` だったりと、命名がばらつく。`src/` の階層を `tests/` 側でもそのまま再現するはずが、いつのまにか構造がずれている。こうした「あるべき構造」は、誰も明文化しないまま少しずつ崩れていく。

コーディングエージェントを使うようになると、この問題がさらに顕在化する。プロンプトで指示しても、エージェントが意図と異なる場所にファイルやディレクトリを作ることがある。人間のレビューだけでこれを毎回拾うのは現実的でない。生成された構造が宣言した「型」に合っているかを機械的に検査できれば、エージェントの逸脱をその場で検出できる。

「各コンポーネントディレクトリには必ずテストファイルを置く」「`hooks/` は各ページのディレクトリの中に置く」といった規約は、どこに書かれているか。多くの場合は CONTRIBUTING.md の片隅か、古い Slack のスレッドか、誰かの頭の中にしかない。

新しいメンバーが加わるたびに口頭で伝え、レビューで指摘し、それでも少しずつ崩れていく。ディレクトリ構造に「型」がないから起きる問題だ。

---

## ディレクトリ構造のモデルは木か森か

私たちはディレクトリ構造を、ルートから枝分かれする 1 本の「木(ツリー)」として思い描きがちだ。だが、複数の木に型を与えて束ねると、それはもう 1 本の木ではない。木のリスト、すなわち「森(forest)」になる。

type-dir が基礎概念に据えているのは、この森のほうだ。1 本の木を検証するのではなく、木の集合(forest)そのものを型として扱う。

---

## type-dir とは何か

ディレクトリ構造の検査ツールである。

`type-dir` は Rust 製の CLI ツールで、YAML で記述したルールファイルをもとに実際のディレクトリツリーを検証し、宣言と食い違いがあれば診断メッセージとして報告する。コードの中身は一切見ず、**構造だけに集中する**。

型システムになぞらえると、`.type-dir.yaml` がディレクトリ構造の「型定義ファイル」にあたる。`type-dir check` がその型検査を実行し、合わなければエラーコードと行番号付きで報告する。

これまで CONTRIBUTING.md の片隅や古い Slack スレッド、あるいは誰かの頭の中にしか存在しなかったディレクトリ構造の規約を、`.type-dir.yaml` という**実行可能な仕様**として記述できる。暗黙知だった規約が、バージョン管理された型定義として誰でも参照、検証できる形になる。
