# `check` コマンドと設定ファイル

## `check` サブコマンド

```console
$ dir-type check [OPTIONS]
```

| オプション | 短縮形 | デフォルト | 説明 |
|---|---|---|---|
| `--config <PATH>` | `-c` | `.dir-type.yaml` | 設定ファイルのパスを指定する |
| `--format <FORMAT>` | なし | `human` | 出力形式。`human` または `json` を指定する |

設定ファイルがカレントディレクトリの `.dir-type.yaml` であれば、引数なしで実行できる。

```console
$ dir-type check
```

検証が通ると何も表示されずに終了コード 0 で終わる。違反があると終了コード 1 で終わる。CI の `&&` チェーンや GitHub Actions の exit-code 判定でそのまま使える。

## 設定ファイルの骨格

設定ファイルは 3 要素で構成される。

```yaml
version: 0       # スキーマバージョン(現在は 0 固定)

entry: root      # 検証の起点となるルール名を指定する

rules:
  - rule: root   # rule: <名前> でルールを定義する
    ::            # :: 以下に、そのディレクトリに存在してよい/しなければならないエントリを列挙する
      - file: README.md
```

`::` は「このディレクトリの中身のルール」を表す。ここに列挙されていないパスがディレクトリ内に存在した場合、`LT001 undeclared path` エラーになる。型システムでいう「余分なフィールドを許さない厳格モード」に相当する。

`::` は YAML として見ると、キー名が `:` のエントリに値を与えているだけだ。JSON でいえば `{ ":": ... }` に相当する。`:`(キー名) + `:`(YAML のキーと値の区切り)が並んで `::` に見えているだけで、ネストしたブロックを簡潔に表すための YAML の記法である。

## JSON 出力で CI と連携する

`--format json` を指定すると、診断情報と検証したディレクトリの走査ログが JSON で stdout に出力される。CI パイプラインで機械的に処理したい場合に使う。

```console
$ dir-type check --format json
```
